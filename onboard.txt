## 1. Project Overview

SitekEats is a multi-vendor food delivery web app with four user roles:
- **BUYER** — customers who order food
- **VENDOR** — restaurants/kitchens that list and manage food
- **RIDER** — delivery personnel
- **ADMIN / SUPPORT_STAFF / FINANCE / OPERATIONS** — internal staff

**Frontend:** Next.js 16+ (App Router), TypeScript strict mode
**Backend:** External REST API built by a separate engineer (Eng. Aubin SIAHA)
**Auth:** Bearer token (JWT) stored in cookies, fetched from `/auth/*` endpoints

## 4. Routing Conventions

### Route groups
- `(auth)` → **removed**. Auth pages now live directly under role folders: `/buyer/login`, `/vendor/login`, etc.
- Global auth pages (forgot, reset) live under `/auth/` — not role-specific
- Account pages live under `/buyer/account/`, `/vendor/dashboard/`, etc.

### Role-based routing
Each role has its own folder under `app/`:
```
/buyer/*        → BUYER pages
/vendor/*       → VENDOR pages
/rider/*        → RIDER pages
/admin/*        → ADMIN pages
/auth/*         → Global auth (forgot, reset, reset-success)
```

### `proxy.ts` (Next.js 16 middleware replacement)
- File: `src/proxy.ts`
- Exported function must be named `proxy` (not `middleware`)
- Guards protected routes by checking `fb_session` and `fb_user_role` cookies
- Current guards:
  - `/buyer/account/*` → requires `fb_session`
  - `/dashboard/vendor` → requires `fb_user_role === 'VENDOR'`
  - `/dashboard/rider` → requires `fb_user_role === 'RIDER'`
  - `/dashboard/admin` → requires `fb_user_role === 'ADMIN'`
  - `/buyer/signup?step=complete` → requires `fb_otp_verified` cookie

## 5. API Layer

### Axios instance (`src/lib/api.ts`)
```ts
import axios from 'axios';
import Cookies from 'js-cookie';

export const api = axios.create({
  baseURL: process.env.NEXT_PUBLIC_API_URL,
  timeout: 12000,
  headers: { 'Content-Type': 'application/json' },
});
```

**Request interceptor** — reads `fb_session` cookie (regular, not HTTP-only — intentional, Axios needs it) and attaches as `Authorization: Bearer <token>`.

**Response interceptor** — handles:
- No response → "Network connectivity lost"
- 401 → clears cookies, redirects to login if on protected route
- 403 → redirects to `/unauthorized`
- Extracts `data.message` from backend error body
- `console.error` in development for all errors

### Endpoints (`src/lib/endpoints.ts`)
```ts
export const API_ROUTES = {
  health: '/',
  auth: {
    otpRequest: '/auth/otp/request',
    otpVerify: '/auth/otp/verify',
    signup: '/auth/signup',
    signin: '/auth/signin',
    profile: '/auth/profile',
    logout: '/auth/logout',
    changePassword: '/auth/change-password',
    changeEmail: '/auth/change-email',
    forgotPassword: '/auth/forgot-password',
    resetPassword: '/auth/reset-password',
  },
} as const;
```

Always add new endpoints here. Never hardcode URL strings in components or fetch functions.

### Feature fetch engines
Each feature has its own `fetchEngine.ts` inside `features/[Feature]/services/`:
```ts
// Pattern
export const authFetchEngine = {
  methodName: async (payload: PayloadType): Promise<ReturnType> => {
    const response = await api.post<ReturnType>(API_ROUTES.auth.endpoint, payload);
    return response.data;
  },
};
```

Rules:
- Every function is fully typed — input payload and return type
- No logic in fetch engine — just HTTP calls
- No toast, no redirect, no side effects here

## 6. Auth System

### Token flow
1. User signs in or signs up
2. Backend returns `{ access_token, refresh_token }`
3. `commitSessionAction` (server action) stores both in cookies
4. Axios reads `fb_session` on every request and attaches as Bearer token
5. `useProfileQuery` fetches `/auth/profile` to get user data

### Cookies

| Cookie | Value | HTTP-only | Set by |
|---|---|---|---|
| `fb_session` | `access_token` | **No** | `commitSessionAction` |
| `fb_refresh_token` | `refresh_token` | **Yes** | `commitSessionAction` |
| `fb_user_role` | decoded role string | **Yes** | `setUserRoleAction` |
| `fb_otp_verified` | contact string | **Yes** | `commitOtpVerifiedAction` |

**Why `fb_session` is NOT HTTP-only:** Axios interceptor runs client-side and needs to read the token to attach it as a Bearer header. HTTP-only cookies cannot be read by JavaScript.

**Why `fb_user_role` IS HTTP-only:** Only `proxy.ts` needs it (server-side edge). JS never needs the role directly — role comes from `useProfileQuery`.

### Server actions (`src/features/Auth/services/serverActions.ts`)
```ts
commitSessionAction(data: TokenResponseData)   // stores access + refresh tokens
destroySessionAction()                          // clears all auth cookies
setUserRoleAction(role: string)                 // stores decoded role
commitOtpVerifiedAction(contact: string)        // stores OTP verified signal (10min TTL)
checkOtpVerifiedAction(): Promise<boolean>      // checks if OTP verified cookie exists
```

### Token decode (`src/lib/token.ts`)
```ts
import { jwtDecode } from 'jwt-decode';

interface TokenPayload {
  role: string;
  sub: string;
  exp: number;
}

export function getRoleFromToken(token: string): string | null {
  try {
    return jwtDecode<TokenPayload>(token)?.role ?? null;
  } catch {
    return null;
  }
}
```

Used immediately after login/signup to decode role and call `setUserRoleAction`.

## 8. TanStack Query Patterns

### No auth context needed
`useProfileQuery` is the global user state. Any component calls it and gets cached data. TanStack Query cache replaces context for server data.

### `useProfileQuery`
```ts
// src/features/Auth/hooks/useProfileQuery.ts
const profileQuery = useQuery<UserProfile, AxiosError>({
  queryKey: ['user-profile'],
  queryFn: authFetchEngine.fetchCurrentProfile,
  enabled: hasToken && isNotOnboarding,  // only runs when logged in
  staleTime: 1000 * 60 * 10,            // 10 minutes
  gcTime: 1000 * 60 * 60,              // 1 hour
  retry: (count, error) => {
    if (error.response?.status === 401 || error.response?.status === 404) return false;
    return count < 2;
  },
  refetchOnWindowFocus: true,
});
```

### `useAuthMutation`
Single hook for all auth mutations. Returns both `mutate` and `mutateAsync` for mutations that need sequential async handling.

**Pattern — redirect stays OUT of mutations:**
```ts
// WRONG — mutation handles redirect
onSuccess: (data) => router.replace('/buyer/account');

// CORRECT — form handles redirect via callback
// In mutation:
onSuccess: () => toast.success('Done');

// In form:
mutation(payload, { onSuccess: () => router.replace('/buyer/account') });
```

**Exception:** `signInMutation` redirects in `onSuccess` because role-based redirect logic belongs there (decodes token, gets role, redirects accordingly).

### Query invalidation pattern
```ts
// After mutation that changes user data
queryClient.invalidateQueries({ queryKey: ['user-profile'] });

// After profile update (optimistic — skip refetch)
queryClient.setQueryData(['user-profile'], updatedData);
```

### Feature query pattern
Every feature has its own hooks folder:
```ts
// features/Orders/hooks/useOrdersQuery.ts
export function useOrdersQuery() {
  return useQuery({
    queryKey: ['orders'],
    queryFn: ordersFetchEngine.fetchOrders,
    staleTime: 1000 * 60 * 2,
  });
}
```

## 10. Design System

### Core rules (non-negotiable)
- **Zero border-radius** — no `rounded-*` anywhere. Sharp corners on everything.
- **No glassmorphism** — no `backdrop-blur`, no frosted glass effects
- **No gradient decorations** — buttons, cards, backgrounds are flat
- **No hover scale transforms** — no `hover:scale-*`
- **No excessive shadows** — `shadow-sm` maximum if ever needed
- **Errors are toasted** — API errors go to Sonner toasts, not inline banners (except auth forms which show inline validation)

### Color usage

|---|---|
| `amber-500` | Primary CTA buttons, active states, accents |
| `amber-50` | Active nav backgrounds, subtle highlights |

| `zinc-500/400` | Secondary text, labels |

| `stone-50` | Page backgrounds |
| `red-500` | Destructive actions (logout confirm button) |

**Amber appears in exactly 3 places per layout:**
1. Active/focused input border

Used only for step transitions in multi-step forms:
```ts
const variants = {
```

No animation on hover, no spring animations, no layout animations. Keep it subtle.

## 11. Buyer Auth Flow

### Login
```
/buyer/login → SignInForm (email or phone + password) → commitSessionAction → decode token → setUserRoleAction → router.replace based on role
```

### Signup (multi-step, single page)
```
/buyer/signup?step=request  → OtpRequestForm → POST /auth/otp/request
/buyer/signup?step=verify   → OtpVerifyForm → POST /auth/otp/verify → commitOtpVerifiedAction
/buyer/signup?step=complete → SignUpDetailsForm (guarded by fb_otp_verified cookie) → POST /auth/signup → commitSessionAction → router.replace('/buyer/account/profile')
```

Step routing: URL param `?step=` drives which form renders. On refresh, correct form shows. Animation slides left (forward) or right (back).

### Forgot / Reset password
```
/auth/forgot → ForgotPasswordForm → POST /auth/forgot-password → shows "link sent" success state
Email/SMS contains link: /auth/reset?token=xyz
/auth/reset → reads token from URL → ResetPasswordForm → POST /auth/reset-password → router.replace('/auth/reset-success')
/auth/reset-success → static page with 3 role login buttons
```

### OTP flow details
- OTP is only for **new user signup** — not for login
- Returning users always use email/password signin
- `fb_otp_verified` cookie: HTTP-only, 10 minute TTL, set after successful OTP verify, deleted on logout

## 13. Component Conventions

### Naming
- Feature components: `PascalCase` in `features/[Feature]/components/`
- Shared primitives: `PascalCase` in `components/ui/`
- Page files: `page.tsx` (Next.js convention)

- Pages that use hooks, state, or browser APIs → `'use client'`

## 14. Toast Conventions (Sonner)

```ts
// Success — standard duration
toast.success('Message', { duration: 4000 });

// Success — important (welcome, account created)
toast.success('Message', { duration: 5000 });

// Error — always longer so user can read
toast.error('Title', { description: error.message, duration: 6000 });

// Info
toast.info('Message', { duration: 4000 });
```

All API `onError` handlers:
1. `console.error` in dev: `if (dev) console.error('[mutationName]', error)`
2. Toast the error with 6000ms duration

Never toast form validation errors — those show inline below inputs.

## 17. Key Decisions Log

| Decision | Reason |
|---|---|
| `fb_session` not HTTP-only | Axios interceptor (client-side) needs to read it for Bearer token |
| Role stored in separate cookie | `proxy.ts` runs at edge before React, needs role without decoding JWT |
| No auth context | TanStack Query cache replaces context for server state |
| Redirect logic in pages not mutations | Mutations are reused across roles — redirect destination varies by role |
| OTP only for signup, not login | Login uses email/password only. OTP is identity verification for new users |
| Separate `TokenResponseData` type | Backend returns no user on login/signup — user always fetched via `/auth/profile` |
| `proxy.ts` not `middleware.ts` | Next.js 16 renamed middleware to proxy |
| Feature-based structure | Scales better than layer-based (components/hooks/pages) as features grow |
